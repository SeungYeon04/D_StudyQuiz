### 1. 소프트웨어 공학

#### 1. UML(Unified Modeling Language) diagram

통합 모델링 언어, 소프트웨어  
시스템을 시각적으로 표현.  
  
> [구조 다이어그램]
> 1. 클래스 다이어그램  
> 정적 관계 설명 / 클래스, 속성, 연산 등 표시  
> 2. 객체 다이어그램  
> 특정 시점의 객체(인스턴스)와 그 관계  
> 3. 컴포넌트 다이어그램  
> 시스템을 구성하는 컴포넌트(모듈)의 구조  
> 4. 배치 다이어그램  
> 시스템의 물리적인 배치  
> 5. 패키지 다이어그램  
> 클래스, 컴포넌트(모듈)를 패키지 단위로 표현  
>  
> [행위 다이어그램]  
> 1. 유스 케이스 다이어그램  
> 사용자와 시스템 기능 간의 상호작용  
> 2. 시퀀스 다이어그램  
> 객체 간 메시지 흐름과 시간 순서 표현  
> 3. 활동 다이어그램  
> 프로그램의 흐름 (조건문, 반복문 등)  
> 4. 상태 다이어그램  
> 객체의 상태 변화 표현  
> 5. 커뮤니케이션 다이어그램  
> 객체 간 메시지 중심  
> 6. 타이밍 다이어그램  
> 시간의 흐름에 따른 객체 상태 표현  

#### 2. 응집도 / 결합도

**[응집도 (Cohesion)]**  
 하나의 모듈 내부 요소들이 얼마나  
밀접하게 연관되어 있는가 (높을수록 좋음)  
  
| 응집도 | 설명 | 
| :--- | :---: | 
| 우연적(Coincidental) | 관련 없음 | 
| 논리적(Logical) | 논리적으로 비슷한 작업(switch-case) | 
| 시간적(Temporal) | 시간적으로 동시에 실행 | 
| 절차적(Procedural) | 절차적으로 연관된 작업들(목적은 다름) | 
| 통신적(Communicational) | 동일한 데이터를 사용하는 작업 |
| 순차적(Sequential) | 한 작업의 결과를 다음 작업 입력 | 
| 기능적(Functional) | 하나의 기능 수행에 집중 |
  
(좋음)우논시절통신기  
  
**[결합도 (Coupling)]**  
모듈 간에 서로 얼마나 의존하는가 (낮을수록 좋음)  
  
| 결합도 | 설명 | 
| :--- | :---: | 
| 내용(Content) | 다른 모듈 내부를 직접 접근하거나 수정 |
| 공통(Common) | 전역 변수를 공유하여 사용 | 
| 외부(External) | 외부 시스템, 인터페이스 등에 의존 |
| 제어(Control) | 제어 정보를 전달 (플래그 등) |
| 스탬프(Stamp) | 필요 이상의 데이터 전달 (a[1]만 필요한데 a 배열 전체 전달) | 
| 자료(Data) | 필요한 데이터만 전달 | 
| 메세지(Message) | 메세지나 이벤트로만 의사소통 | 
  
내공외제스자메(좋음)  
  
### 3. GoF(Ganf of Four) 디자인 패턴 

1. 생성(Creational)패턴  
| 패턴 | 설명 | 
| :--- | :---: | 
| Singleton | 하나의 객체(인스턴스)만 생성 | 
| Factory Method | 상위 클래스가 인터페이스 제공, 하위클래스에서 객체 생성 | 
| Abstract Factory | 관련된 객체군 생성 | 
| Builder | 복잡한 객체를 단계별로 생성 | 
| Prototype | 기존 객체를 복사하여 새 객체 생성 | 
    
2. 구조(Structural)패턴  
| 패턴 | 설명 | 
| :--- | :---: | 
| Observer | 상태 변화 감지(유튜브 구독, 알림설정 > 구독 영상에 뜬다, 알림이 온다) |  
| Strategy | 알고리즘을 런타임에 동적으로 선택 (최신순, 좋아요순, 조회순) | 
| Command | 요청을 객체로 캡슐화하여 요청자와 수행자를 분리 | 
| State | 객체 상태에 따라 동작 변경 (캐릭터: 걷기, 뛰기) | 
| Chain of Responsibility | 요청을 처리할 객체를 링크드 리스트처럼 연결 | 
| Interpreter | 언어나 문법을 해성하는 인터프리터 구현 | 
| Iterator | 컬렉션 내부 구조를 노출하지 않고 순회, 접근이 잦은 객체에 동일한 인터페이스 제공 | 
| Mediator | 객체 간 상호작용을 중재자 객체가 관리 | 
| Memento | 객체 상태를 저장하고 복원 (UNDO기능:ctrl+z) | 
| Template Method | 상위 클래스에 기본 로직, 하위 클래스에 세부 구현(extends) | 
| Visitor | 객체의 구조는 유지(수정불가)한 상태로 외부에서 동작 정의 | 
  
### 4. 커버리지 (Coverage)
코드가 얼마나 테스트 되었는가.  
  
1. 문장(Statement) 커버리지  
- 정의: 프로그램 내의 모든 문장이 적어도 한 번 실행되었는가  
- 장점: 구현이 쉬움  
- 단점: 조건문 내 각 분기의 결과는 확인하지 않음  
```
if(x > 0 && y < 10) {
    y = 1; // 이 부분이 실행되면 통과 
} 
```
  
2. 결정(Branch) 커버리지 (분기 커버리지)  
- 정의: 모든 분기 즉, if, switch 등에서 True/False의 양 경로를 모두 통과했는가  
```
if(x > 0 && y < 10) { // 조건문의 결과로 True/False 2가지 모두 테스트하면 통과 
    y = 1; 
} 
```
  
3. 조건(Condition) 커버리지  
- 정의: 하나의 조건식에 포함된 각 조건 (AND, OR로 이루어진)의 개별 결과에서 True/False가 모두 실행되었는가  
```
if(x > 0 && y < 10) { // 개별 조건에서 True/False 실행되면 통과 
    y = 1; //T/F 
} //F/T 이렇게 2번만 테스트 해도 통과 - 결과는 신경 쓰지 않음. 
```
  
4. 조건/결정(Condition/Decision) 커버리지  
- 정의: 분기 결과(결정) + 각 조건식의 결과를 모두 테스트  
    - 조건 전체 결과와 각 조건이 모두 True/False 실행 되었는가  
  
| A | B | 결과 |
| :--- | :--- | :--- | 
| T | F | T | 
| F | T | F | 
이것만 해도 통과  
  
5. 변형 조건/결정 커버리지(MC/DC: Modified Condition/Decision Coverage)  
- 정의: 복합 조건식에서 각 조건이 독립적으로전체 결과에 영향을 주는가  
  - 그러니가 개별 조건의 True/False 변화가 결과에 영향을 미치는 것만 확인하면 통과  
```
if(A||B){}  
```
| A | B | 결과 |
| :--- | :--- | :--- | 
| T | X | T | 
| F | T | T | 
| F | F | F | 
  
6. 다중 조건(Multiple Condition) 커버리지  
- 정의: 복합 조건에서 모든 가능한 조건 조합 테스트  
  - 조건이 n개면 `2<sup>n</sup>`번의 테스트 필요  
```
if(A||B){}  
```
| A | B | 결과 |
| :--- | :--- | :--- | 
| T | T | T | 
| F | T | F | 
| T | F | F | 
| F | F | F | 
  
7. 경로(Path) 커버리지  
- 정의: 프로그램 내 가능한 모든 실행 경로 테스트  
  - 예제 확인 필요  
  
### 2. 데이터베이스(DataBase Management System) 

#### 1. 정규화 (Normalization) 

데이터의 중복을 제거하여 Database의 무결성을 유지  

1. 제1 정규형  
테이블의 컬럼이 원자값을 갖는다  
  
2. 제2 정규형  
부분 종속 제거 (기본키가 2개 이상 컬럼의 조합일 때, 그중 일부에만 종속인 행 제거)  
  
3. 제3 정규형  
이행적 종속 제거 (A > B이고 B > C일 때, A > C인 경우를 제거)  
  
4. BCNF(Boyce-Codd Normal Form)  
제3 정규형 + 모든 결정자가 후보키  
  
5. 제4 정규형  
BCNF + 다치 종속(Multi-Valued Dependency) 제거  
  
| 학생 | 자격증 | 외국어 |
| :--- | :--- | :--- | 
| A | 정보처리 | 영어 | 
| A | 정보처리 | 일본어 | 
| A | 컴활 | 영어 | 
| A | 컴활 | 일본어 | 
  
여기서 자격증과 외국어는 서로 독립적  
  
| 학생 | 자격증 |
| :--- | :--- |
| A | 컴활 |
| A | 정보처리 |
  
| 학생 | 외국어 |
| :--- | :--- |
| A | 영어 |
| A | 일본어 |
  
#### 2. 반정규화/역정규화 

시스템 성능 개선 및 개발, 유지, 보수의 편의성을 위해  
의도적으로 정규화 규칙을 위배하는 것  
  
#### 3. 무결성 제약 조건 (Integrity Constraint) 

1. 개체 무결성 제약 조건 (Entity)  
- 기본키는 NULL을 가질 수 없다.  
- 중복되는 값을 가질 수 없다.  
  
2. 참조 무결성 제약 조건 (Referential)  
- 외래키는 다른 테이블의 기본키를 참조해야 한다.  
  
3. 도메인 무결성 제약 조건 (Domain)  
- 속성(Attribute) 값은 도메인(각 컬럼 값의 범위)에 속해야 한다.  
  
#### 키의 종류 

- 기본키: 테이블에서 튜플을 유일하게 식별하는 하나 이상의 속성  
- 후보키: 테이블에서 튜플을 유일하게 식별할 수 있는 최소 속성 집합  
- 대체키: 후보키 중, 기본키로 선택되지 않은 키  
- 외래키: 다른 테이블의 기본키를 참조하는 속성  
- 슈퍼키: 최소성을 보장하지 않는 식별자  
- 합성키: 둘 이상의 속성을 결합하여 유일한 튜플을 식별하는 키  
  
#### SQL 기본 명령어 

1. DDL(Data Definition Language) 데이터 정의어  
- CREATE : 데이터베이스, 테이블 생성  
- ALTER : 테이블 구조 변경  
- DROP : 데이터베이스, 테이블 삭제  
- TRUNCATE : 테이블 데이터 전체 삭제  
  
2. DCL(Data Control Language) 데이터 제어어  
- GRANT : 권한 부여  
- REVOKE : 권한 취소  
- COMMIT : 트랜잭션 확정  
- ROLLBACK : 트랜잭션 취소  
  
3. DCL (Data Control Language) 데이터 제어어  
- 행: row, tuple, cardinality(개수)  
- 열: column, attribute, degree(개수, 차수)  
- table, relation  
- 릴레이션 키 = 기본키(Primary Key)  
- 릴레이션 스키마 : 릴레이션의 구조(속성들의 집합)  
- 릴레이션 인스턴스 : 어느 시점에 릴레이션에 들어있는 튜플들의 집합  
  
### 3. 네트워크 

1. OSI 7 Layer와 TCP/IP  
  
**전송계층**  
| TCP | UCP |
| :--- | :--- |
| 연결형 | 비연결형 |
| 신뢰성 O | 신뢰성 X |
| 느림 | 빠름 | 
| 혼잡제어 O | 혼잡제어 X | 
| 흐름제어 O | 흐름제어 X | 
  
**네트워크 계층/인터넷 계층**  
- IP(Internet Protocol) : 각 장치의 논리 주소(IP 주소)를 이용하여 데이터를 목적지까지 라우팅  
- ICMP(Internet Control Message Protocol) : 네트워크 오류 메세지 전달  
- ARP(Address Resolution Protocol) : IP 주소에 해당하는 MAC 주소를 찾는 프로토콜  
- RARP(Reverse Address Resolution Protocol) : MAC 주소를 IP 주소로 바꾸는 프로토콜  
  
2. IP 주소와 서브넷팅  
- prefix: /26 등  
- 특수 주소  
  - (ex) IP: 196.168.10.25 / 서브넷 마스크 255.255.255.0 > prefix : 24  
  - 196.168.10.25/24  
    - 네트워크 주소: 196.168.10.0  
    - 브로도 캐스트 주소: 196.168.10.255  
    - 가용 범위: 196.168.10.1 ~ 196.168.10.254  
  
3. 라우팅 프로토콜  
  
> 1. 내부 라우팅 프로토콜 (IGP : Interior Gateway Protocol) 
> - RIP(Routing Information Protocol) : 거리 백터 방식 (최대 hop: 15)  
> - OSPF(Open Shortest Path First) : 링크 상태 방식 (Dijkstra 알고리즘)  
> - EIGRP, IGRP : Cisco 전용  
>
> 2. 외부 라이팅 프로토콜 (EGP : External Gateway Protocol)  
> - BGP(Border Dateway Protocol) : 경로 벡터 방식 (AS 간 라우팅)  
  
4. 통신 방식  

| 명칭 | 연결상태 | 속도 | 패킷 순서 | 예시 | 
| :--- | :--- | :--- | :--- | :--- |
| 데이터그램 | 비연결형 | 빠름 | 보장X | UDP |
| 가상회선 | 연결형 | 느림 | 보장O | TCP |
  
5. 통신 장비  

- 허브 vs 스위치 (전송단위)  
  - 허브: 브로도캐스트  
  - 스위치: MAC 주소 테이블 기반 > 대상에게만 전달  
- 스위치 vs 라우터 (내외부)  
  - 스위치: 같은 네트워크 내에서 MAC 주소 기준 프레임 전달  
  - 라우터: 다른 네트워크 간 IP 주소 기준으로 패킷 전달  
- 라우터 vs 게이트웨이 (프로토콜)  
  - 라우터: 같은 프로토콜(IP) 내의 경로 설정  
  - 게이트웨이: 서로 다른 프로토콜 간 변환
